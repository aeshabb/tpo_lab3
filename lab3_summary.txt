# Лабораторная работа 3: Автоматизированное тестирование UI сложных веб-приложений
## Цель работы: 
Протестировать функциональность веб-приложения (GitHub) с использованием различных подходов в рамках экосистемы Selenium (генерация шаблонов в Selenium IDE и их выполнение на языке Java с использованием старого API Selenium RC).

---

## Выбранные прецеденты (Use Cases) для тестирования:

В рамках тестирования реализовано 3 сценария (прецедента), охватывающих разные типы взаимодействия с UI:

1. **Поиск репозиториев (Search Repository)**
   - Действия: Пользователь открывает главную страницу, находит строку поиска, кликает на нее или вводит запрос "Selenium WebDriver". Важное условие: обход Rate Limit в браузере Firefox при помощи прямых навигаций вместо посимвольного введения текста.
   - Ожидаемый результат: Загрузка страницы со списком результатов поиска, успешный переход на первый репозиторий в списке.

2. **Просмотр тем (Topics Overview)**
   - Действия: Открытие раздела "Topics" (`/topics`), чтение заголовка раздела. Клик по первой теме из предложенного списка.
   - Ожидаемый результат: Переход на страницу соответствующей темы, проверка URL (наличие `/topics/`).

3. **Фильтрация трендовых репозиториев (Trending Filter) - ** [НОВЫЙ ПРЕЦЕДЕНТ]
   - Описание: Тестирование работы с кастомными выпадающими списками и фильтрами на платформе.
   - Действия: Пользователь заходит на страницу трендов (`/trending`), открывает выпадающее меню языков ("Spoken Language" / "Language"), и выбирает элемент "Java".
   - Ожидаемый результат: Страница перезагружается, меняется контент и URL (`/trending/java`), отображая трендовые репозитории по выбранному языку. 
   - Сложность: Перехват кликов всплывающими окнами (Shadow DOM / Стили GitHub). Меню реализовано как тег `<summary>` внутри `<details>`, для которого требуются особенные JS-сценарии или точный XPath.

> *Все прецеденты запускаются параметризованными тестами JUnit 5 (аннотация `@ParameterizedTest` совместно с `@ValueSource(strings = {"chrome", "firefox"})`) с проверкой на обеих указанных платформах.*

## Архитектура и методы тестирования: "Под капотом"

В этой лабораторной работе мы не используем современный подход Selenium WebDriver напрямую. Согласно заданию, использован подход **Selenium RC (Remote Control)** API. Чем это обусловлено и в чем разница?

### 1. Подход: Современный Selenium 4 WebDriver
В современном Selenium архитектура работает так:
- Мы пишем код на Java (`driver.findElement(By.xpath(...)).click()`).
- WebDriver сериализует команду по протоколу **W3C WebDriver** во внутренний формат и отправляет ее по сети (HTTP) локальному сервису-драйверу (ChromeDriver / GeckoDriver).
- Драйвер непосредственно командует браузером **на глубоком уровне самого движка** (перехватывая управление через CDP(Chrome DevTools Protocol) или BiDi).
- **Плюсы**: Очень высокая скорость, работа в "нативном" слое, возможность работы с любым сложным DOM-деревом.

### 2. Подход: Старый Selenium IDE + Selenium RC API (Требования ЛР)
Selenium RC (Selenium 1.x) — это устаревшая архитектура, которая работала совершенно иначе.
- RC Server запускался как прокси и **внедрял JavaScript-код** (Selenium Core) напрямую в тестируемую веб-страницу.
- Вместо работы с движком браузера, команды RC (команды из старого IDE типа `selenium.click("xpath=...")`) превращались в JS-события: создавалось синтетическое DOM-событие "MouseClick", которое отправлялось нужному элементу силами самого языка Javascript браузера.
- **Минусы**: Огромные проблемы с современным вебом (React, Shadow DOM, сложные перекрывающиеся слои CSS). Внедренный JS не умел кликать по элементу, если он был, например, перекрыт прозрачным всплывающим окном или стилем `z-index`. 

### Как старый RC работает с новыми драйверами в нашем коде?
Поскольку оригинальный сервер Selenium RC более не работает с современными версиями браузеров, библиотека Selenium предоставляет класс-адаптер **`WebDriverBackedSelenium`**.
Он выступает в роли "Моста времени":
1. Вы вызываете старую RC команду: `selenium.click(locator)`.
2. `WebDriverBackedSelenium` перехватывает этот вызов.
3. Он "переводит" старый синтаксис (например `"xpath=..."`) в современные объекты локаторов `By`.
4. Внутренне он вызывает современный метод `WebDriver` или внедряет Javascript через `JavascriptExecutor`, пытаясь эмулировать старое поведение RC. 

### Почему пришлось писать `runScript` или обходные маневры?
Современный интерфейс GitHub блокирует или скрывает элементы (например, строку поиска или кнопки в меню). Из-за того, что мы обязаны использовать устаревший интерфейс классического `Selenium` и его команды (простые `click()`, `type()`), которые транслируются из старого синтаксиса с потерями, мы оборачиваем их в конструкции `try...catch`:
```java
try {
   selenium.click("xpath=//summary[contains(., 'Language')]");
} catch (Exception e) {
   // RC-прослойка не смогла пробиться сквозь слои CSS современного React/GitHub.
   // Тогда мы вспоминаем истинную суть Selenium RC (внедрение скриптов руками):
   selenium.runScript("document.evaluate(\"//summary[...\", ...).click();");
}
```

### Решение проблемы Rate Limits (Too many requests) в Firefox:
Мы столкнулись с тем, что старый способ введения текста `selenium.type()` в браузере Firefox генерировал мгновенную лавину запросов к API-автодополнения GitHub. Сайт очень быстро расценивал это как DDoS атаку и блокировал доступ.
Решение под капотом (как мы договорились с сервером): 
1. Использование объекта `FirefoxOptions` с флагами сокрытия драйвера от системы обнаружения ботов (отключение `dom.webdriver.enabled = false` и подмена `User-Agent`).
2. Замена посимвольного ввода на прямой переход по сгенерированному поисковому URI через команду `selenium.open(/search?q=...)`. Уход от `selenium.type()` решил проблему перегрузки API GitHub ложными запросами автодополнения в старом интерфейсе.

## Итоги
В результате работы, все прецеденты были успешно экспортированы из шаблонов Selenium IDE, перенесены на язык Java при помощи архаичного синтаксиса `com.thoughtworks.selenium.*`, и адаптированы под суровую реальность современных динамических SPA-сайтов при помощи гибридного подхода: старого синтаксиса и вспомогательного выполнения скриптовых "инъекций" (`runScript`).
Все 3 комплексных теста (Search, Topics, Trending) стабильно выполняются для браузеров Google Chrome и Mozilla Firefox через JUnit 5.
